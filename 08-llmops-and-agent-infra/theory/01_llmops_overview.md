# 从 MLOps 到 LLMOps：大模型应用的生命周期管理

## 1. 引言：为什么需要 LLMOps

### 1.1 传统 MLOps 回顾

传统 MLOps 围绕以下生命周期展开：

```
数据收集 → 特征工程 → 模型训练 → 模型验证 → 部署 → 监控 → 迭代
```

核心关注点：
- **数据管道**：ETL、特征存储、数据版本化
- **训练管道**：超参搜索、分布式训练、实验追踪
- **部署管道**：模型打包、服务化、A/B 测试
- **监控**：数据漂移、模型性能退化

### 1.2 LLMOps 的本质区别

LLM 应用的生命周期与传统 ML 有根本性差异：

| 维度 | 传统 MLOps | LLMOps |
|------|-----------|--------|
| 模型来源 | 自己训练 | 调用 API / 微调基座模型 |
| 核心资产 | 训练数据 + 模型权重 | Prompt + 检索管道 + 编排逻辑 |
| 迭代对象 | 模型参数 | Prompt / RAG 配置 / Agent 策略 |
| 评估方式 | 精确指标（AUC/F1） | 多维度模糊评估（相关性/忠实度/有用性） |
| 部署单元 | 单一模型 | 复合系统（LLM + Retriever + Tools） |
| 成本模型 | GPU 训练时长 | Token 消耗 + 延迟 |
| 失败模式 | 预测偏差 | 幻觉、Prompt 注入、上下文丢失 |

## 3. Prompt 管理

### 3.1 Prompt 即代码

在 LLMOps 中，Prompt 是第一等公民：

```python
# 反模式：硬编码 Prompt
response = llm.chat("帮我总结这篇文章：" + article)

# 正确做法：Prompt 模板化 + 版本控制
class PromptRegistry:
    """Prompt 注册中心 - 类比微服务的配置中心"""
    
    def __init__(self, backend="git"):  # 或 "database"
        self.backend = backend
        self.cache = {}
    
    def get_prompt(self, name: str, version: str = "latest") -> PromptTemplate:
        """获取指定版本的 Prompt 模板"""
        key = f"{name}:{version}"
        if key not in self.cache:
            self.cache[key] = self._load_from_backend(name, version)
        return self.cache[key]
```


## 4. 模型选择策略

### 4.1 决策矩阵

```
                    能力需求
            低 ─────────────────── 高
         ┌──────────────────────────┐
    低   │  本地小模型              │  GPT-4o / Claude Opus
    │    │  (Qwen2-7B)             │  (复杂推理)
    延   │                          │
    迟   ├──────────────────────────┤
    要   │  本地中型模型            │  混合路由
    求   │  (Qwen2-72B on H20)     │  (简单→小模型,复杂→大模型)
    │    │                          │
    高   └──────────────────────────┘
```

### 4.2 H20 GPU 上的模型部署规划

8 张 H20 (96GB × 8 = 768GB)可承载：

| 模型 | 参数量 | 所需 GPU | 吞吐量 (tokens/s) |
|------|--------|----------|-------------------|
| Qwen2.5-7B | 7B | 1 | ~2000 |
| Qwen2.5-72B | 72B | 4 (TP=4) | ~400 |
| Llama3-70B | 70B | 4 (TP=4) | ~350 |
| DeepSeek-V2-236B | 236B (MoE) | 8 (TP=8) | ~200 |

### 4.3 路由策略

```python
class ModelRouter:
    """智能模型路由 - 根据请求复杂度选择模型"""
    
    def __init__(self):
        self.models = {
            "simple": "qwen2.5-7b",      # 简单查询
            "medium": "qwen2.5-72b",     # 一般推理
            "complex": "claude-opus",     # 复杂推理
        }
    
    def route(self, query: str, context: dict) -> str:
        complexity = self._estimate_complexity(query, context)
        if complexity < 0.3:
            return self.models["simple"]
        elif complexity < 0.7:
            return self.models["medium"]
        else:
            return self.models["complex"]
```


## 8. 实践建议

### 8.1 从后端工程师视角理解 LLMOps

| API 版本控制 | Prompt 版本控制 |

### 8.2 渐进式采纳路径

```
阶段 1（1-2 周）：
  → 单一 LLM API 调用
  → 基础 Prompt 模板
  → 简单评估脚本

阶段 2（3-4 周）：
  → RAG 系统
  → 向量数据库

阶段 3（5-6 周）：
  → Agent 系统

  → 企业级平台

  → 持续评估 + 自动迭代
```

## 9. 本模块实践路线

本模块的 10 个 Lab 和 1 个企业级项目覆盖了 LLMOps 的完整生命周期。建议：

1. **先理论后实践**：每个 Lab 对应一个理论章节
2. **渐进式构建**：每个 Lab 的产出将成为最终项目的组件
3. **评估驱动**：每个 Lab 都包含评估步骤
4. **生产视角**：始终考虑可观测性、可扩展性、容错性

## 10. 小结

LLMOps 不是对传统 MLOps 的简单扩展，而是一套全新的工程实践体系。核心思维转变：

- 从"训练模型"到"编排系统"
- 从"精确指标"到"多维评估"
- 从"静态部署"到"持续迭代"
- 从"单一模型"到"复合 Agent"

对于有微服务经验的工程师，很多 LLMOps 概念都有对应的后端概念。关键是理解 LLM 的非确定性本质带来的工程挑战，并用系统化的方法来管理这种不确定性。
