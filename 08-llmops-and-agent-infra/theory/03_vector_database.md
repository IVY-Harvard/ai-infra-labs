# 向量数据库

## 1. 为什么需要专用向量数据库

### 1.1 向量检索的本质

在 RAG 系统中，文本通过 Embedding 模型转为高维向量（如 1024 维），检索的本质是在高维空间中找到与查询向量最相似的 K 个向量。

**精确搜索的问题**：
- 暴力计算：O(n × d)，n=文档数，d=维度
- 1 亿文档 × 1024 维 = 每次查询需计算 1000 亿次浮点运算
- 不可接受的延迟

**近似最近邻（ANN）**：用少量精度换取数量级的速度提升

### 1.2 何时需要向量数据库

```
数据规模          推荐方案
< 10K 文档  →    内存 ANN (FAISS/Annoy) 足够
10K - 1M    →    轻量级向量数据库 (Chroma/Milvus Lite)
1M - 100M   →    分布式向量数据库 (Milvus/Qdrant)
> 100M      →    定制化方案 (Milvus 集群 + 定制索引)
```

## 2. 核心索引算法

### 2.1 HNSW（Hierarchical Navigable Small World）

**原理**：构建多层跳表式的图结构

```
Layer 3:  ●─────────────────────●  (最稀疏，用于快速定位区域)
          │                     │
Layer 2:  ●────●────────●───────●  (中等密度)
          │    │        │       │
Layer 1:  ●──●─●──●──●──●──●───●  (最密集，精确搜索)
          │  │ │  │  │  │  │   │
Layer 0:  ●●●●●●●●●●●●●●●●●●●●●  (全部数据点)
```

**搜索过程**：
1. 从最高层开始，贪心搜索找到局部最近点
2. 向下层移动，继续贪心搜索
3. 在底层（Layer 0）做精确的邻居搜索

**关键参数**：
- `M`：每个节点的最大连接数（默认 16，越大越精确但越慢）
- `ef_construction`：构建时的搜索宽度（默认 200）
- `ef_search`：查询时的搜索宽度（默认 50，可动态调整）

**特点**：
- 优势：查询速度快 O(log n)，精度高（recall > 95%）
- 劣势：内存占用大（存储图结构），构建时间长
- 适用：中等规模 + 高精度要求

### 2.2 IVF（Inverted File Index）

**原理**：先聚类再搜索

```
┌─────────────────────────────────────┐
│      原始向量空间                     │
│                                       │
│   ┌─────┐  ┌─────┐  ┌─────┐        │
│   │ C1  │  │ C2  │  │ C3  │  ...   │
│   │●●●●●│  │●●●● │  │●●●●●│        │
│   │ ●●● │  │●●●●●│  │ ●●● │        │
│   └─────┘  └─────┘  └─────┘        │
│                                       │
│   训练阶段：K-Means 聚类得到质心      │
│   查询阶段：先找最近的 nprobe 个簇    │
│             再在这些簇内暴力搜索      │
└─────────────────────────────────────┘
```

**关键参数**：
- `nlist`：聚类中心数（通常 sqrt(n) 到 4*sqrt(n)）
- `nprobe`：查询时访问的簇数（越大越精确但越慢）

**特点**：
- 优势：内存效率高，支持增量更新
- 劣势：需要训练阶段，小规模数据表现不佳
- 适用：大规模数据 + 内存受限

### 2.3 PQ（Product Quantization）

**原理**：向量压缩 — 将高维向量分段量化

```
原始向量 (1024 维, float32):  4096 bytes
                    ↓
分成 128 个子空间，每个 8 维
每个子空间量化为 1 字节 (256 个聚类中心)
                    ↓
压缩后: 128 bytes (压缩比 32:1)
```

**IVF + PQ 组合**（最常用的大规模方案）：
```
IVF 负责粗筛 → PQ 负责存储 + 近似距离计算 → 可选 Refine 阶段用原始向量精排
```

### 2.4 算法选型指南

| 场景 | 推荐算法 | 理由 |
|------|----------|------|
| < 100 万，追求精度 | HNSW | 最佳精度/速度平衡 |
| 100 万 - 1 亿 | IVF_HNSW | 结合两者优势 |
| > 1 亿，内存受限 | IVF_PQ | 极低内存占用 |
| 需要 GPU 加速 | IVF_FLAT (GPU) | FAISS GPU 原生支持 |
| 流式数据 | HNSW | 支持增量插入 |

## 3. 主流向量数据库对比

### 3.1 Milvus

```
定位：云原生分布式向量数据库
架构：存算分离，组件化设计

┌─────────────────────────────────────────┐
│                 Milvus 架构              │
│                                          │
│  ┌─────────┐  ┌─────────┐              │
│  │  Proxy  │  │  Proxy  │  (接入层)    │
│  └────┬────┘  └────┬────┘              │
│       └──────┬─────┘                    │
│       ┌──────▼──────┐                   │
│       │ Coordinator │  (协调层)         │
│       │ Root│Query│Data│Index           │
│       └──────┬──────┘                   │
│       ┌──────▼──────┐                   │
│       │   Workers   │  (执行层)         │
│       │ Query│Data│Index               │
│       └──────┬──────┘                   │
│       ┌──────▼──────┐                   │
│       │   Storage   │  (存储层)         │
│       │ MinIO/S3 + etcd + MQ           │
│       └─────────────┘                   │
└─────────────────────────────────────────┘

优势：
  • 水平扩展能力强
  • 支持多种索引类型
  • GPU 索引支持 (CAGRA)
  • 丰富的过滤表达式
  • 活跃的社区生态

适用：大规模生产环境、需要分布式部署
```

### 3.2 Qdrant

```
定位：高性能 Rust 实现的向量数据库
架构：单机为主，支持分布式

优势：
  • Rust 实现，性能优异
  • Payload 过滤功能强大
  • 支持多向量（同一文档的不同 Embedding）
  • 量化支持好（Scalar/PQ/Binary）
  • REST + gRPC 双协议
  • 部署简单，运维成本低

适用：中等规模、重视过滤性能、追求运维简单
```

### 3.3 Weaviate

```
定位：AI-native 向量数据库
架构：Go 实现，模块化

优势：
  • 内置向量化模块（自动调用 Embedding API）
  • GraphQL 查询接口
  • 多模态支持（图像/音频）
  • Hybrid Search 原生支持
  • 多租户支持好

适用：需要内置向量化、多模态场景
```

### 3.4 Pinecone

```
定位：全托管向量数据库服务
架构：Serverless

优势：
  • 零运维
  • 自动扩缩容
  • 全球多区域
  • Sparse-Dense 混合搜索

劣势：
  • 供应商锁定
  • 数据出境问题
  • 成本不透明

适用：海外业务、不想运维、快速验证
```

### 3.5 pgvector

```
定位：PostgreSQL 向量搜索扩展
架构：PostgreSQL 插件

优势：
  • 复用现有 PG 基础设施
  • 事务支持
  • 与关系数据同库
  • SQL 接口，学习成本低
  • HNSW + IVFFlat 索引

劣势：
  • 大规模性能不及专用向量数据库
  • 缺乏分布式能力
  • 内存管理受 PG 限制

适用：中小规模、已有 PG 基础设施、需要事务一致性
```

### 3.6 对比总结

| 维度 | Milvus | Qdrant | Weaviate | Pinecone | pgvector |
|------|--------|--------|----------|----------|----------|
| 规模上限 | 100亿+ | 10亿 | 10亿 | 10亿+ | 1000万 |
| 查询延迟 | <10ms | <5ms | <10ms | <20ms | <50ms |
| 过滤能力 | 强 | 很强 | 中 | 中 | 很强(SQL) |
| 分布式 | 原生 | 支持 | 支持 | 托管 | 需要Citus |
| 运维复杂度 | 高 | 中 | 中 | 无 | 低 |
| 生态/SDK | 丰富 | 丰富 | 丰富 | 丰富 | SQL |
| 开源 | ✓ | ✓ | ✓ | ✗ | ✓ |
| GPU 加速 | ✓ | ✗ | ✗ | ✗ | ✗ |

## 4. 选型决策树

```
开始
  │
  ├─ 数据量 > 1 亿？
  │    ├─ YES → Milvus 集群
  │    └─ NO  ↓
  │
  ├─ 已有 PostgreSQL 且规模 < 500 万？
  │    ├─ YES → pgvector
  │    └─ NO  ↓
  │
  ├─ 需要复杂过滤 + 高性能？
  │    ├─ YES → Qdrant
  │    └─ NO  ↓
  │
  ├─ 需要内置向量化 / 多模态？
  │    ├─ YES → Weaviate
  │    └─ NO  ↓
  │
  ├─ 不想运维 + 海外业务？
  │    ├─ YES → Pinecone
  │    └─ NO  ↓
  │
  └─ 默认推荐 → Qdrant (中小规模) / Milvus (大规模)
```

## 5. 生产环境最佳实践

### 5.1 索引配置

```python
# Milvus HNSW 生产配置
index_params = {
    "index_type": "HNSW",
    "metric_type": "COSINE",
    "params": {
        "M": 32,                  # 连接数，生产建议 16-64
        "efConstruction": 256,    # 构建质量，生产建议 128-512
    }
}

search_params = {
    "metric_type": "COSINE",
    "params": {
        "ef": 128,  # 查询精度，根据 recall 要求动态调整
    }
}
```

### 5.2 数据管理

```
最佳实践：
├── 元数据设计
│   • 为每个向量存储：source_id, chunk_index, timestamp, version
│   • 支持按时间范围/来源过滤
│
├── 数据更新策略
│   • 增量更新：新文档直接插入
│   • 全量更新：版本化 collection，蓝绿切换
│   • 失效策略：标记删除 + 定期压缩
│
├── 多租户
│   • 小租户：Partition Key 隔离
│   • 大租户：独立 Collection
│
└── 备份恢复
    • 定期快照
    • 跨区域复制
    • 灾备演练
```

### 5.3 性能优化

```
查询优化：
├── 预过滤 vs 后过滤
│   • 预过滤：先过滤再搜索（过滤条件选择性高时）
│   • 后过滤：先搜索再过滤（过滤条件选择性低时）
│
├── 批量查询
│   • 批处理多个查询，提升吞吐
│   • 适合离线评估场景
│
├── 缓存策略
│   • 热点查询缓存（Semantic Cache）
│   • 向量 + 结果双层缓存
│
└── 资源配置
    • 内存 > 索引大小 × 1.5
    • SSD 用于磁盘索引（DiskANN）
    • 网络带宽：分布式部署时关键
```

### 5.4 监控指标

| 指标 | 含义 | 告警阈值 |
|------|------|----------|
| query_latency_p99 | 查询延迟 | > 100ms |
| recall@10 | 召回率 | < 0.90 |
| index_build_time | 索引构建时间 | > 1h |
| memory_usage | 内存使用率 | > 85% |
| qps | 每秒查询数 | 接近压测极限 80% |
| segment_count | Segment 数量 | 过多需压缩 |

## 6. H20 GPU 加速向量检索

### 6.1 GPU 索引方案

```python
# Milvus GPU 索引（NVIDIA CAGRA）
# H20 的 96GB HBM3 可缓存大量向量数据
gpu_index_params = {
    "index_type": "GPU_CAGRA",
    "metric_type": "L2",
    "params": {
        "intermediate_graph_degree": 64,
        "graph_degree": 32,
    }
}

# 适用场景：
# - 超大规模实时查询（10M+ 向量，<5ms 延迟）
# - 批量向量检索（离线评估）
# - 与 LLM 推理共享 GPU 资源
```

### 6.2 混合部署策略

```
H20 × 8 资源分配建议：
  GPU 0-3: LLM 推理 (72B 模型, TP=4)
  GPU 4:   Embedding 模型服务
  GPU 5:   Reranker 模型服务
  GPU 6:   GPU 向量索引 (热数据)
  GPU 7:   备用/弹性
```

## 7. 小结

向量数据库选型的核心考量：

1. **规模**：决定是否需要分布式方案
2. **延迟**：决定索引算法和硬件配置
3. **过滤**：复杂过滤需求影响选型
4. **运维**：团队能力和成本预算
5. **生态**：与现有技术栈的集成程度

对于本模块的学习环境（8 × H20），推荐：
- 学习阶段：Milvus Lite（单机内嵌）或 Qdrant Docker
- 生产阶段：Milvus 集群 + GPU 索引
