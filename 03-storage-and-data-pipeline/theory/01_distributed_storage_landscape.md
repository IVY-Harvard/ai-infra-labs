# 01 — AI 存储全景：从 NFS 到新型 AI 存储

## 为什么存储是 AI 基础设施的隐形瓶颈

你可能已经习惯了这样的工作流：把模型文件放在 NFS 上，训练脚本挂载后直接读取。在单机 8 卡场景下这通常没问题，但当你遇到以下场景时，存储就成了真正的瓶颈：

```
场景 1：8 卡同时加载 70B 模型 → NFS 带宽打满，加载时间从 30s 变成 5min
场景 2：多节点训练每 1000 步写 Checkpoint → 64 卡同时突发写 200GB，NFS 直接卡死
场景 3：训练数据集 10TB 小文件 → ls 一个目录要 30 秒，DataLoader 大量时间浪费在元数据操作
```

理解 AI 工作负载对存储的特殊要求，是选择正确存储方案的基础。

## AI 工作负载的三种典型 IO 模式

### 模式一：大文件顺序读 — 模型加载

```
特征：
- 文件大小：1GB ~ 400GB（单个模型权重文件）
- 访问模式：顺序读，一次性读完
- 并发度：多 GPU 同时读同一文件
- 关键指标：吞吐量（GB/s）
- 典型场景：模型加载、Checkpoint 恢复

实际数据（以 Llama-2-70B 为例）：
- 模型大小：约 140GB（FP16）
- 8 卡加载：每卡读 ~17.5GB（张量并行切分后）
- NFS 1Gbps：17.5GB / 125MB/s ≈ 140s
- NFS 10Gbps：17.5GB / 1.25GB/s ≈ 14s
- 本地 NVMe SSD：17.5GB / 3.5GB/s ≈ 5s
```

### 模式二：小文件随机读 — 训练数据加载

```
特征：
- 文件大小：1KB ~ 10MB（图片、文本片段）
- 访问模式：随机读，高并发
- 并发度：多 Worker 并行读取不同文件
- 关键指标：IOPS（每秒 IO 操作数）
- 典型场景：图片分类训练、多模态训练

痛点分析：
- 1M 张 256KB 图片 = 256GB 数据
- 每 epoch 需要遍历全部文件
- 4 个 DataLoader Worker × 8 GPU = 32 并发读
- NFS 元数据性能：~1000 IOPS → 32 并发 = 每 Worker 仅 31 IOPS
- 实际瓶颈不在带宽，在于元数据和 IOPS
```

### 模式三：突发大块写 — Checkpoint 持久化

```
特征：
- 文件大小：10GB ~ 500GB（完整训练状态）
- 访问模式：突发写入，周期性
- 并发度：所有 GPU 同时写
- 关键指标：写吞吐 + 写延迟
- 典型场景：每 N 步保存 Checkpoint

写入量估算（以 8 卡 Llama-2-70B 微调为例）：
- 模型参数：140GB
- 优化器状态：280GB（Adam: 2× 参数量）
- 梯度：140GB
- 总计：~560GB 需要在几十秒内写完
- 要求写带宽：560GB / 30s ≈ 18.7 GB/s
```

## 三大存储类别对比

### 第一类：POSIX 文件系统

#### NFS（Network File System）

```
架构：客户端-服务器模型，单一服务器提供文件服务
优点：
  ✓ 部署简单，Linux 原生支持
  ✓ 完全 POSIX 兼容
  ✓ 适合小规模共享
缺点：
  ✗ 单服务器瓶颈（带宽/IOPS 都受限于单台机器）
  ✗ 元数据和数据都在同一服务器
  ✗ 不适合大规模并发访问

典型性能（单 NFS 服务器，10GbE）：
  - 顺序读带宽：~1.1 GB/s
  - 随机 4K IOPS：~50,000
  - 元数据操作（stat/open）：~5,000 ops/s
```

#### Lustre

```
架构：分离式元数据和数据，数据分条到多个 OST（Object Storage Target）
组件：
  - MDS（Metadata Server）：管理文件元数据
  - OSS（Object Storage Server）：存储实际数据
  - Client：POSIX 挂载
优点：
  ✓ 超高聚合带宽（可达 TB/s 级别）
  ✓ 数据自动分条（striping）
  ✓ HPC 领域久经考验
缺点：
  ✗ 运维复杂度极高
  ✗ 元数据单点（MDS 性能瓶颈）
  ✗ 小文件场景性能一般

典型性能（中等规模集群）：
  - 顺序读带宽：10-100+ GB/s（取决于 OST 数量）
  - 随机 4K IOPS：~200,000（分散到多 OST）
  - 元数据操作：~10,000 ops/s（MDS 瓶颈）
```

#### GPFS / Spectrum Scale

```
架构：分布式并行文件系统，所有节点可同时访问
优点：
  ✓ 真正的并行访问，无单一元数据瓶颈
  ✓ 高级特性丰富（快照、配额、分级存储）
  ✓ IBM 企业级支持
缺点：
  ✗ 昂贵的商业许可
  ✗ 需要专业运维团队
  ✗ 与特定硬件绑定较深

典型性能：
  - 顺序读带宽：50-500 GB/s
  - 随机 4K IOPS：~500,000+
  - 元数据操作：~100,000 ops/s
```

### 第二类：对象存储

#### S3 / MinIO

```
架构：RESTful API，数据以对象（Key-Value）形式存储，天然分布式
S3 API 已成为事实标准，MinIO 是最流行的自建方案

优点：
  ✓ 无限扩展，不存在容量瓶颈
  ✓ 部署简单（MinIO 单二进制文件）
  ✓ 生态丰富，几乎所有工具都支持 S3 API
  ✓ 低成本（可用 HDD，支持纠删码节省空间）
缺点：
  ✗ 不支持 POSIX 语义（不能 open/seek/write）
  ✗ 延迟较高（HTTP 开销，首字节延迟 10-100ms）
  ✗ 小文件性能差（每个对象一次 HTTP 请求）
  ✗ 不支持追加写和原地修改

MinIO 典型性能（4 节点，NVMe）：
  - GET 吞吐：35 GB/s
  - PUT 吞吐：20 GB/s
  - 小对象 GET：~50,000 ops/s
```

### 第三类：新型 AI 存储

#### VAST Data

```
定位：专为 AI/ML 工作负载设计的统一存储
架构：NFS/S3 双协议，全闪存，全局去重和压缩
核心特性：
  - DASE（Disaggregated, Shared-Everything）架构
  - QLC Flash 降低成本
  - 同时支持 POSIX 和 S3 访问
  - 针对 GPU Direct Storage 优化
适用场景：预算充足，需要统一存储平台
```

#### WEKA

```
定位：高性能并行文件系统
架构：分布式、全闪存优化
核心特性：
  - 亚毫秒级延迟
  - 线性扩展带宽
  - 支持 GPU Direct Storage (GDS)
  - 可以分层到 S3
性能数据：
  - 单客户端读带宽：100+ Gbps
  - 元数据操作：>1M ops/s
适用场景：对性能要求极高的 AI 训练
```

#### JuiceFS

```
定位：开源的 POSIX 兼容分布式文件系统
架构：元数据引擎（Redis/TiKV/MySQL）+ 对象存储后端 + 本地缓存
核心特性：
  ✓ 完全开源，社区版免费
  ✓ POSIX 兼容，任何应用零修改适配
  ✓ 元数据和数据分离，各自独立扩展
  ✓ 本地 SSD 缓存加速热数据访问
  ✓ 支持 Kubernetes CSI Driver
缺点：
  ✗ 性能受限于后端对象存储和元数据引擎
  ✗ 写性能不如原生 Lustre
  ✗ 需要额外维护元数据引擎

适用场景：中小规模 AI 团队，需要低成本高兼容性方案
```

## AI 存储选型决策树

```
Q1: 你的预算级别？
├── 高预算（可采购商业存储）
│   ├── 需要极致性能 → WEKA
│   ├── 需要统一 NFS/S3 → VAST Data
│   └── 已有 IBM 生态 → GPFS
├── 中等预算（可自建集群）
│   ├── HPC 经验丰富 → Lustre
│   ├── 需要 POSIX + S3 → JuiceFS
│   └── 纯对象存储场景 → MinIO
└── 低预算（利用现有基础设施）
    ├── 单机/少量节点 → NFS + 本地 SSD 缓存
    └── 已有 S3 → JuiceFS + S3 后端

Q2: 你的核心需求？
├── 模型加载速度 → 本地 NVMe + 缓存层（JuiceFS/Alluxio）
├── 训练数据吞吐 → Lustre/WEKA/GPFS（高聚合带宽）
├── Checkpoint 写入 → 本地 SSD 暂存 + 异步上传到对象存储
└── 成本优化 → MinIO + JuiceFS（SSD 缓存热数据）
```

## 你的 8 卡 H20 环境建议

```
当前环境：8 × H20 GPU，单节点，NFS 挂载模型文件

短期优化（零成本）：
1. 模型文件拷贝到本地 NVMe SSD，消除 NFS 加载瓶颈
2. Checkpoint 先写本地 SSD，再异步同步到 NFS
3. 训练数据打成大文件（tar/WebDataset），减少小文件 IOPS 压力

中期方案（低成本）：
1. 部署 MinIO 作为本地对象存储
2. 部署 JuiceFS（Redis + MinIO），获得 POSIX 兼容 + 本地 SSD 缓存
3. 配置 JuiceFS 缓存策略，热数据自动缓存到 NVMe

长期扩展（多节点时）：
1. JuiceFS + TiKV（元数据）+ MinIO/Ceph（数据）
2. 或评估 WEKA/VAST Data 等商业方案
```

## 本章小结

- AI 工作负载有三种典型 IO 模式，分别对应不同的存储需求
- 没有万能的存储方案，选型取决于预算、规模和核心需求
- JuiceFS 是中小团队的最佳起步方案：开源、POSIX 兼容、本地缓存加速
- 将「大文件顺序读」「小文件随机读」「突发大块写」三种模式分开优化，是存储优化的核心思路
