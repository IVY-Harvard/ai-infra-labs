# 02 — JuiceFS 架构详解

## 为什么选 JuiceFS

在上一章的存储全景中，我们看到 JuiceFS 对中小 AI 团队非常友好：

```
核心卖点：
1. 开源免费（Apache 2.0 协议）
2. 完全 POSIX 兼容 — 现有代码零修改
3. 元数据和数据分离 — 各自独立扩展
4. 本地 SSD 缓存 — 热数据接近本地 NVMe 性能
5. Kubernetes CSI 原生支持 — 云原生适配
6. 后端灵活 — S3/MinIO/Ceph/OSS/COS 等 20+ 后端
```

你的场景：8 张 H20 GPU，当前用 NFS 挂载。JuiceFS 能让你在不改代码的情况下获得：
- 模型加载速度提升 3-5x（通过本地缓存）
- Checkpoint 写入不再阻塞训练
- 未来多节点扩展零成本

## JuiceFS 三层架构

```
┌─────────────────────────────────────────────────────────┐
│                    应用层（POSIX/FUSE）                    │
│         PyTorch / DeepSpeed / 任何 Linux 应用              │
└──────────────────────────┬──────────────────────────────┘
                           │ open/read/write/close
┌──────────────────────────▼──────────────────────────────┐
│                   JuiceFS Client                          │
│  ┌─────────────┐  ┌─────────────┐  ┌─────────────────┐ │
│  │  FUSE 接口   │  │  缓存管理    │  │  数据分块/合并   │ │
│  └─────────────┘  └─────────────┘  └─────────────────┘ │
└────────┬────────────────────────────────────┬───────────┘
         │ 元数据操作                          │ 数据操作
         │ (stat/mkdir/rename)                │ (read/write blocks)
         ▼                                    ▼
┌─────────────────┐              ┌─────────────────────────┐
│  元数据引擎      │              │    对象存储后端           │
│                 │              │                         │
│  · Redis        │              │  · MinIO (自建)          │
│  · TiKV         │              │  · AWS S3               │
│  · MySQL/PG     │              │  · 阿里云 OSS            │
│  · etcd         │              │  · Ceph RADOS           │
│  · SQLite(单机)  │              │  · ...                  │
└─────────────────┘              └─────────────────────────┘
```

### 第一层：元数据引擎

元数据引擎负责管理所有的文件系统结构信息：

```
元数据包含：
- 文件/目录的名称和层级关系
- 文件大小、权限、时间戳
- 文件到数据块（Chunk/Slice）的映射关系
- 锁信息（并发访问控制）

元数据引擎选型：
┌────────────┬──────────────┬──────────────┬──────────────────┐
│ 引擎       │ 适用规模      │ 性能         │ 运维复杂度         │
├────────────┼──────────────┼──────────────┼──────────────────┤
│ Redis      │ 1亿文件以下   │ 最高         │ 中（需主从/哨兵）  │
│ TiKV       │ 10亿文件+    │ 高           │ 高（PD+TiKV集群） │
│ MySQL/PG   │ 5000万文件   │ 中           │ 中                │
│ etcd       │ 1000万文件   │ 中高         │ 中                │
│ SQLite     │ 100万文件    │ 低（单机）    │ 极低（零运维）     │
└────────────┴──────────────┴──────────────┴──────────────────┘

AI 场景推荐：
- 单机开发/测试：SQLite（零依赖）
- 单节点 8 卡生产：Redis（性能最优，百万级文件足够）
- 多节点集群：TiKV（水平扩展，支持海量文件）
```

### 第二层：对象存储后端

数据以固定大小的块（Block）存储在对象存储中：

```
数据分块策略：
- Chunk 大小：64 MB（默认）
- Block 大小：4 MB（默认，上传到对象存储的单元）
- 一个 Chunk 由多个 Block 组成

示例：写入一个 200MB 的模型文件
  → 拆分为 4 个 Chunk（3×64MB + 1×8MB）
  → 每个 Chunk 进一步拆为 Block
  → 每个 Block 独立上传到对象存储

对象存储中的 Key 命名：
  {volume_name}/chunks/{chunk_id}/{block_id}
```

### 第三层：本地缓存

这是 JuiceFS 在 AI 场景的核心竞争力：

```
缓存层级：
┌────────────────────────────────────────────────┐
│           内存缓存（Kernel Page Cache）           │
│           容量：系统可用内存                       │
│           延迟：~100ns                           │
├────────────────────────────────────────────────┤
│           JuiceFS 客户端内存缓存                  │
│           容量：可配置（默认 300MB）               │
│           延迟：~1μs                             │
├────────────────────────────────────────────────┤
│           本地磁盘缓存（SSD/NVMe）               │
│           容量：可配置（如 500GB）                 │
│           延迟：~100μs                           │
├────────────────────────────────────────────────┤
│           远端对象存储                            │
│           容量：无限                              │
│           延迟：~10ms                            │
└────────────────────────────────────────────────┘

缓存策略：
- 读缓存：读取的数据自动缓存到本地 SSD
- 预读取：顺序读时自动预读后续数据块
- 缓存淘汰：LRU（最近最少使用）
- 缓存预热：支持手动预热指定目录/文件
```

## JuiceFS 读写流程详解

### 读操作流程

```
应用调用 read(fd, buf, 1MB, offset=100MB)
    │
    ▼
[1] JuiceFS Client 查询本地缓存
    │
    ├── 命中（本地 SSD 有缓存）
    │   └── 直接返回数据，延迟 ~100μs ✓
    │
    └── 未命中
        │
        ▼
[2] 查询元数据引擎：获取 offset=100MB 对应的 Chunk/Block ID
        │
        ▼
[3] 从对象存储下载对应 Block
        │
        ▼
[4] 数据写入本地缓存（下次读直接命中）
        │
        ▼
[5] 返回数据给应用
```

### 写操作流程

```
应用调用 write(fd, data, 10MB)
    │
    ▼
[1] JuiceFS Client 将数据写入客户端缓冲区（内存）
    │  → 此时 write() 立即返回（异步写入）
    │
    ▼
[2] 数据积累到一个 Block（4MB）后，上传到对象存储
    │
    ▼
[3] 上传成功后，更新元数据引擎（记录新 Block 位置）
    │
    ▼
[4] 文件关闭 / fsync 时，确保所有数据持久化

写入特性：
- 默认异步写入（writeback 模式），高吞吐
- 支持同步写入（--writeback=false），强一致性
- 写缓冲区可配置大小（--buffer-size，默认 300MB）
```

## 为什么 JuiceFS 适合 AI 场景

### 场景 1：模型加载

```
传统 NFS：
  8 GPU 同时从 NFS 读 Llama-70B (140GB)
  → NFS 带宽 1.2GB/s → 需要 ~117 秒

JuiceFS + 本地 SSD 缓存：
  首次加载：从对象存储下载 → 缓存到本地 NVMe
  → 约 60 秒（10Gbps 网络到对象存储）
  后续加载：直接从本地 NVMe 读取
  → 140GB / 3.5GB/s = 40 秒
  且 8 GPU 各自从本地缓存读，无共享瓶颈！

关键命令（预热模型到缓存）：
  juicefs warmup /mnt/jfs/models/llama-70b/
```

### 场景 2：训练数据读取

```
传统方案（百万小文件在 NFS 上）：
  - ls 目录：30 秒（元数据瓶颈）
  - DataLoader 随机读：受限于 NFS IOPS

JuiceFS 方案：
  - 元数据在 Redis：ls 目录 < 1 秒
  - 数据缓存到本地 SSD：随机读 IOPS 接近本地磁盘
  - 建议：将数据打包为 WebDataset tar 格式，转随机读为顺序读
```

### 场景 3：Checkpoint 写入

```
传统 NFS：
  8 卡写 60GB Checkpoint → NFS 被打满 → 训练暂停 50 秒

JuiceFS（writeback 模式）：
  8 卡写到本地缓冲区 → write() 立即返回 → 训练几乎不停
  后台异步上传到对象存储 → 最终一致性
  
配置：
  juicefs mount ... --writeback --buffer-size 4096
  # 4GB 写缓冲，足够缓冲一次 Checkpoint
```

## JuiceFS 快速部署（你的 H20 环境）

```bash
# 1. 安装 JuiceFS
curl -sSL https://d.juicefs.com/install | sh -

# 2. 部署 Redis（元数据引擎）
docker run -d --name redis-jfs \
  -p 6379:6379 \
  -v /data/redis:/data \
  redis:7-alpine redis-server --appendonly yes

# 3. 部署 MinIO（对象存储后端）
docker run -d --name minio-jfs \
  -p 9000:9000 -p 9001:9001 \
  -v /data/minio:/data \
  -e MINIO_ROOT_USER=minioadmin \
  -e MINIO_ROOT_PASSWORD=minioadmin \
  minio/minio server /data --console-address ":9001"

# 4. 创建 MinIO bucket
mc alias set local http://localhost:9000 minioadmin minioadmin
mc mb local/jfs-data

# 5. 格式化 JuiceFS 卷
juicefs format \
  --storage minio \
  --bucket http://localhost:9000/jfs-data \
  --access-key minioadmin \
  --secret-key minioadmin \
  redis://localhost:6379/1 \
  ai-storage

# 6. 挂载（带本地 SSD 缓存）
juicefs mount \
  redis://localhost:6379/1 \
  /mnt/jfs \
  --cache-dir /nvme-ssd/jfs-cache \
  --cache-size 512000 \       # 500GB 缓存
  --buffer-size 4096 \        # 4GB 写缓冲
  --prefetch 3 \              # 预读取 3 个 Block
  --writeback \               # 异步写入
  -d                          # 后台运行
```

## 关键配置参数调优

```bash
# AI 模型加载优化
--cache-size 512000          # 缓存越大越好，装得下整个模型最佳
--prefetch 3                 # 顺序读时预取后续 Block
--max-uploads 50             # 并行上传线程数（Checkpoint 写入时有用）
--max-deletes 10             # 并行删除线程数

# Checkpoint 写入优化
--writeback                  # 写入先到本地，异步上传（关键！）
--buffer-size 4096           # 写缓冲 4GB
--upload-delay 0             # 关闭后立即上传（默认就是 0）

# 训练数据读取优化
--cache-dir /nvme/jfs-cache  # 用最快的 NVMe SSD
--free-space-ratio 0.1       # SSD 保留 10% 空间
--cache-partial-only false   # 缓存完整 Block（非部分）

# 监控
juicefs stats /mnt/jfs       # 实时性能统计
juicefs profile /mnt/jfs     # IO 追踪
```

## 本章小结

- JuiceFS 的三层架构（元数据引擎 + 对象存储 + 本地缓存）天然适合 AI 场景
- 元数据引擎选 Redis（单节点）或 TiKV（多节点）
- 本地 NVMe SSD 缓存是性能的关键：模型加载和数据读取都受益
- writeback 模式 + 大 buffer 让 Checkpoint 写入对训练影响最小化
- 相比 Ceph 和 Lustre，JuiceFS 部署和运维成本低一个量级
