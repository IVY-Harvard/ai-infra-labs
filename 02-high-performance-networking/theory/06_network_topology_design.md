# AI 集群网络拓扑设计

## 概述

网络拓扑决定了集群中任意两个 GPU 之间的通信带宽和延迟上限。不同的拓扑设计在成本、性能、扩展性之间做出不同取舍。对于 8×H20 集群，选择合适的拓扑需要同时考虑并行策略（TP/DP/PP）对网络的不同需求。

---

## Fat-Tree 拓扑

### 经典三层架构

```
           ┌─────────────┐  ┌─────────────┐
           │  Core Switch │  │  Core Switch │     核心层
           └──┬──┬──┬──┬─┘  └─┬──┬──┬──┬──┘
              │  │  │  │       │  │  │  │
         ┌────┘  │  │  └──┐ ┌─┘  │  │  └────┐
         ▼       ▼  ▼     ▼ ▼    ▼  ▼       ▼
      ┌──────┐ ┌──────┐ ┌──────┐ ┌──────┐
      │Agg-1 │ │Agg-2 │ │Agg-3 │ │Agg-4 │    聚合层
      └┬──┬──┘ └┬──┬──┘ └┬──┬──┘ └┬──┬──┘
       │  │     │  │     │  │     │  │
       ▼  ▼     ▼  ▼     ▼  ▼     ▼  ▼
      ToR ToR  ToR ToR  ToR ToR  ToR ToR       接入层 (ToR)
       |   |    |   |    |   |    |   |
      ─┴─ ─┴─  ─┴─ ─┴─  ─┴─ ─┴─  ─┴─ ─┴─     计算节点
```

### 非阻塞 vs 阻塞设计

**非阻塞 (Non-Blocking, 1:1)**：
- 上行带宽 = 下行带宽
- 任意节点对之间均可获得全线速
- 成本最高：需要大量核心交换机和上行线缆

**阻塞 (Blocking)**：
- 上行带宽 < 下行带宽
- 带宽收敛比 = 下行带宽 : 上行带宽

```
示例: ToR 交换机 32 端口 (400G)

非阻塞 1:1 设计:
├── 16 端口 → 下行连接 16 个节点 (16 × 400G = 6.4T)
└── 16 端口 → 上行连接聚合层   (16 × 400G = 6.4T)
收敛比 = 1:1, 跨 ToR 通信不降速

阻塞 2:1 设计:
├── 24 端口 → 下行连接 24 个节点 (24 × 400G = 9.6T)
└──  8 端口 → 上行连接聚合层   ( 8 × 400G = 3.2T)
收敛比 = 3:1, 跨 ToR 通信降速到 1/3

阻塞 3:1 设计 (不推荐用于 AI 训练):
├── 28 端口 → 下行
└──  4 端口 → 上行
收敛比更高, 跨 ToR 通信严重受限
```

### 带宽收敛比选型

| 收敛比 | 成本 | 适用场景 |
|--------|------|---------|
| 1:1 | 最高 | TP 并行跨节点、大模型训练 |
| 2:1 | 中等 | 纯 DP 训练、中等通信量 |
| 3:1 | 较低 | 推理服务、通信量小的任务 |

**8×H20 建议**：AI 训练网络建议 1:1 或最多 2:1。

## Dragonfly 拓扑

### 架构设计

Dragonfly 是一种低直径、低成本的拓扑，由 Google TPU Pod 推广：

```
Dragonfly 二级结构:

Group 0          Group 1          Group 2
┌─────────┐     ┌─────────┐     ┌─────────┐
│ S─S─S─S │ ←→  │ S─S─S─S │ ←→  │ S─S─S─S │
│ | | | | │     │ | | | | │     │ | | | | │
│ N N N N │     │ N N N N │     │ N N N N │
└─────────┘     └─────────┘     └─────────┘
     ↑                                ↑
     └────────────────────────────────┘
     
S = 交换机, N = 计算节点
组内: 全互连 (每个交换机到组内所有其他交换机)
组间: 每组分配部分端口连接其他组 (全局链路)
```

### 特点

- **低直径**：最多 3 跳到达任意节点 (组内1跳 + 组间1跳 + 目标组内1跳)
- **低成本**：全局链路数远少于 Fat-Tree，线缆成本低
- **自适应路由关键**：组间链路可能拥塞，必须支持自适应路由
- **适用规模**：数千到数万节点

### 与 Fat-Tree 对比

| 维度 | Fat-Tree | Dragonfly |
|------|----------|-----------|
| 跳数 (最坏) | 4-6 跳 | 3 跳 |
| 线缆数量 | 多 (O(N√N)) | 少 (O(N^(4/3))) |
| 带宽均匀性 | 高 (非阻塞设计时) | 依赖路由算法 |
| 故障容忍 | 多条冗余路径 | 组间链路故障影响大 |
| 运维复杂度 | 低 (层次清晰) | 中 (路由策略复杂) |

## Rail-Optimized 拓扑

### NVIDIA DGX SuperPOD 方案

Rail-Optimized 是 NVIDIA 为 GPU 集群优化的拓扑，核心思想是**按 GPU 编号对齐网络连接**：

```
传统连接 (每节点所有 NIC 连同一 ToR):

节点 0: [GPU0 GPU1 ... GPU7] ──8×NIC──→ ToR-0
节点 1: [GPU0 GPU1 ... GPU7] ──8×NIC──→ ToR-1
问题: 节点间 AllReduce 流量全部经过 ToR 上行链路，成为瓶颈

Rail-Optimized 连接 (每个 GPU 编号对应独立网络):

Rail 0 交换机: 节点0-GPU0, 节点1-GPU0, 节点2-GPU0 ...
Rail 1 交换机: 节点0-GPU1, 节点1-GPU1, 节点2-GPU1 ...
...
Rail 7 交换机: 节点0-GPU7, 节点1-GPU7, 节点2-GPU7 ...
```

### 可视化对比

```
=== Rail-Optimized 拓扑 ===

 Rail Switch 0    Rail Switch 1    ...    Rail Switch 7
      |                |                       |
  ┌───┼───┐        ┌───┼───┐              ┌───┼───┐
  │   │   │        │   │   │              │   │   │
Node0-GPU0  Node0-GPU1               Node0-GPU7
Node1-GPU0  Node1-GPU1               Node1-GPU7
Node2-GPU0  Node2-GPU1               Node2-GPU7
  ...         ...                      ...

优势:
- 相同编号 GPU 间通信只需 1 跳 (同一 Rail Switch)
- AllReduce 天然分散到 8 个独立网络，无带宽争抢
- 每条 Rail 可独立配置和排障
```

### 适用条件

- 节点内 GPU 数量固定 (如 8 GPU/节点)
- 每个 GPU 有独立 NIC
- AllReduce 是主要通信模式 (DP 训练)
- 集群规模适中 (单 Rail Switch 可容纳所有节点)

### Rail-Optimized 的限制

- **TP 并行不友好**：TP 需要节点间不同编号 GPU 通信，跨 Rail 需上行
- **扩展性受限**：单 Rail Switch 端口数 = 最大节点数
- **故障影响**：单 Rail Switch 故障影响所有节点的对应 GPU

## 网络拓扑与并行策略

### 不同并行策略的网络需求

```
Tensor Parallelism (TP):
├── 通信模式: AllReduce (小消息、高频)
├── 带宽需求: ★★★★★ (极高)
├── 延迟需求: ★★★★★ (极敏感)
└── 最佳拓扑: 节点内 NVLink/PCIe P2P；跨节点需 1:1 非阻塞

Data Parallelism (DP):
├── 通信模式: AllReduce (大消息、每步一次)
├── 带宽需求: ★★★☆☆ (中等，可 overlap)
├── 延迟需求: ★★☆☆☆ (不太敏感)
└── 最佳拓扑: Rail-Optimized；2:1 收敛可接受

Pipeline Parallelism (PP):
├── 通信模式: Point-to-Point (激活值传递)
├── 带宽需求: ★★★☆☆ (中等)
├── 延迟需求: ★★★★☆ (影响 Pipeline Bubble)
└── 最佳拓扑: 低跳数拓扑；PP 阶段放在网络临近的节点

Expert Parallelism (EP, MoE):
├── 通信模式: All-to-All (每 token 路由到不同 Expert)
├── 带宽需求: ★★★★☆ (高，且不规则)
├── 延迟需求: ★★★★☆ (高)
└── 最佳拓扑: 非阻塞 Fat-Tree；All-to-All 对带宽均匀性要求极高
```

### 8×H20 拓扑设计建议

#### 小规模 (2-8 节点，16-64 GPU)

```
推荐: 单层 IB 交换机直连

   Quantum-2 NDR Switch (64 端口)
   ├── Port 1-2:  Node0 (双端口 bond)
   ├── Port 3-4:  Node1
   ├── ...
   └── Port 15-16: Node7

优点: 1 跳延迟，满带宽，配置简单
限制: 最多 32 节点 (双端口) 或 64 节点 (单端口)
成本: 1 台交换机 + NDR 线缆
```

#### 中规模 (8-64 节点，64-512 GPU)

```
推荐: 两层 Fat-Tree (Leaf-Spine)

    Spine Switch 0    Spine Switch 1    Spine Switch 2    Spine Switch 3
         |                 |                 |                 |
    ┌────┼────┐       ┌────┼────┐       ┌────┼────┐       ┌────┼────┐
    │    │    │       │    │    │       │    │    │       │    │    │
  Leaf0  Leaf1  Leaf2  Leaf3  Leaf4  Leaf5  Leaf6  Leaf7

每个 Leaf 连接 8 个节点 (下行) + 4 个 Spine (上行)
收敛比: 配置为 1:1 (下行端口数 = 上行端口数)
```

#### 大规模 (64+ 节点，512+ GPU)

```
推荐: 三层 Fat-Tree 或 Rail-Optimized + Fat-Tree 混合

Rail-Optimized (DP 通信):
├── 8 条独立 Rail 网络
├── 每条 Rail: Leaf-Spine 两层
└── 相同编号 GPU 间通信 1-2 跳

辅助 Fat-Tree (TP/PP 通信):
├── 传统 Leaf-Spine
├── 非阻塞设计
└── 处理跨 GPU 编号的通信
```

## 实际设计考量

### 线缆规划

| 线缆类型 | 距离 | 成本 | 适用 |
|----------|------|------|------|
| DAC 铜缆 | ≤ 2m | 低 | 机架内节点到 ToR |
| AOC 有源光缆 | ≤ 100m | 中 | 机架间、ToR 到 Spine |
| 光模块+光纤 | ≤ 10km | 高 | 跨楼/跨园区 |

### 容错设计

```
关键原则:
1. 消除单点故障 (SPOF)
   - 双 Spine 交换机互为备份
   - 节点双 NIC 连不同 Leaf
   - Subnet Manager 主备部署

2. 优雅降级
   - 单链路故障: 自适应路由绕行，带宽降低但不中断
   - 单 Leaf 故障: 影响范围限于该 Leaf 下的节点
   - 单 Spine 故障: 全网带宽按比例降低
```

### 监控指标

```bash
# 关键监控项
# 1. 端口错误计数
perfquery <lid> <port>  # 检查 SymbolError, LinkRecover, RcvErrors

# 2. 带宽利用率
ibstat | grep Rate      # 确认链路速率正确
# 通过 UFM 或 ibdiagnet 获取流量统计

# 3. 拥塞信号
# IB: 查看 VL 仲裁统计
# RoCE: 监控 PFC 暂停帧频率 (应接近零)

# 4. 路由表一致性
ibroute <switch_lid>    # 检查线性转发表
```

## 小结

网络拓扑设计需要在成本、性能、扩展性之间平衡。对于 8×H20 集群：小规模首选单交换机直连（简单高效）；中规模选择两层 Fat-Tree（成熟可靠）；大规模考虑 Rail-Optimized 方案（针对 DP 训练优化）。核心原则——TP 通信要求极低延迟和极高带宽（尽量放在节点内），DP 通信对带宽需求中等（可通过 Rail-Optimized 优化），收敛比应根据实际通信模式而非一刀切。
