# 多集群 GPU 调度方案对比

## 1. 方案概览

```
┌────────────────────────────────────────────────────────────────────┐
│                  多集群 GPU 调度方案对比                              │
├──────────────┬─────────────┬────────────────┬─────────────────────┤
│    方案      │   类型       │   GPU 调度支持  │   适用规模          │
├──────────────┼─────────────┼────────────────┼─────────────────────┤
│ KubeFed      │ 联邦控制面   │  基础          │  2-10 集群          │
│ Admiralty    │ 虚拟调度     │  好            │  2-20 集群          │
│ Liqo         │ 资源共享     │  好            │  2-10 集群          │
│ Karmada      │ 联邦调度     │  好            │  10-100 集群        │
│ OCM          │ 开放集群管理  │  中            │  10+ 集群           │
│ 自建方案     │ 自定义       │  可定制        │  任意规模           │
└──────────────┴─────────────┴────────────────┴─────────────────────┘
```

## 2. KubeFed (Kubernetes Federation v2)

```
架构：
┌─────────────────────────────────────────────────┐
│           Federation Control Plane               │
│  ┌───────────────────────────────────────────┐  │
│  │  FederatedDeployment / FederatedJob       │  │
│  │  → 声明要在哪些集群部署什么               │  │
│  └───────────────────────────────────────────┘  │
└──────────────────────┬──────────────────────────┘
                       │ 同步
         ┌─────────────┼─────────────┐
         ▼             ▼             ▼
   ┌──────────┐  ┌──────────┐  ┌──────────┐
   │ Cluster A │  │ Cluster B │  │ Cluster C │
   │ 8x H20   │  │ 4x A100  │  │ Cloud GPU │
   └──────────┘  └──────────┘  └──────────┘

特点：
  - 模板化：定义一次，分发到多集群
  - 调度策略：按权重/偏好选择集群
  - 状态聚合：收集各集群的工作负载状态

局限：
  - 项目维护不活跃
  - 不支持动态调度决策（基于实时 GPU 可用性）
  - GPU 资源感知较弱
```

## 3. Karmada (推荐方案)

```
架构：
┌────────────────────────────────────────────────────┐
│              Karmada Control Plane                   │
│                                                     │
│  ┌────────────────┐  ┌────────────────────────┐   │
│  │ karmada-       │  │  Scheduling 策略:       │   │
│  │ controller-    │  │  - Cluster 资源感知     │   │
│  │ manager        │  │  - GPU 可用性检查       │   │
│  │                │  │  - 亲和性/反亲和性      │   │
│  └────────────────┘  └────────────────────────┘   │
│                                                     │
│  CRDs:                                             │
│  - PropagationPolicy (分发策略)                     │
│  - OverridePolicy (集群特定覆盖)                    │
│  - ResourceBinding (资源绑定)                       │
└────────────────────────────────────────────────────┘

GPU 调度示例：
  PropagationPolicy:
    placement:
      clusterAffinity:
        clusterNames: [onprem-h20, cloud-a100]
      replicaScheduling:
        type: Divided
        weightPreference:
          staticWeightList:
          - targetCluster: onprem-h20
            weight: 2           # 优先私有集群
          - targetCluster: cloud-a100
            weight: 1           # 不够时 burst 到云上
```

## 4. Admiralty（虚拟节点方案）

```
原理：在源集群创建虚拟节点，代表目标集群的资源

源集群 (Host)                    目标集群 (Target)
┌─────────────────────┐          ┌─────────────────────┐
│                     │          │                     │
│  真实节点: gpu-node │          │  真实节点: gpu-node │
│  虚拟节点: cloud-*  │ ───────► │  实际执行 Pod       │
│                     │  调度到   │                     │
│  kube-scheduler     │  虚拟节点  │                     │
│  正常调度，不感知   │  时转发   │                     │
│  多集群             │          │                     │
└─────────────────────┘          └─────────────────────┘

优势：
  - 对用户透明（不需要多集群 API）
  - 复用 K8s 原生调度
  - 虚拟节点可以上报 GPU 资源数量

劣势：
  - 跨集群网络延迟
  - 调试复杂
```

## 5. GPU 调度决策维度

```
多集群 GPU 调度决策：

输入：
  - GPU 请求量和类型（H20/A100/H100）
  - 数据位置（哪个集群有训练数据）
  - 成本约束（Spot vs On-Demand）
  - 延迟要求（推理需要低延迟）
  - 可用性（各集群当前 GPU 空闲量）

决策逻辑：
  1. 数据本地性优先：训练数据在哪，任务就在哪
  2. 成本最优：Spot 有空就用 Spot，按需实例作为 fallback
  3. 能力匹配：需要 NVLink → 选 H100/H20，不需要 → A100 够用
  4. 容量规划：超出本地配额 → burst 到云上
```

## 6. 对你的 8x H20 环境的建议

```
当前（单集群）：
  → 不需要多集群调度
  → 用 Kueue 做好队列管理即可

扩展路径：
  1. 第一步：加一个 Cloud 集群用于 burst
     → 用 Karmada + PropagationPolicy
     → 私有集群优先，不够时自动 burst

  2. 第二步：多区域推理部署
     → 训练在私有集群
     → 推理服务用 Karmada 分发到多区域

  3. 第三步：Spot 实例利用
     → 实验性训练跑在 Spot GPU 上
     → 配合容错（checkpoint + 自动恢复）
```
